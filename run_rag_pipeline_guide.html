🚀 실습 가이드

실습 가이드 run rag pipeline

HTML/MD를 사이드카 JSON으로 처리 → OpenSearch 인덱싱 → 검색까지 한 스크립트로

📄 HTML 처리 📝 MD 처리 🏦 OpenSearch 인덱싱 🔍 하이브리드 검색

📊 한눈에 보기 — 전체 흐름

전제조건 체크부터 검색 테스트까지 7단계로 이루어져요.

💡
편한 방법: 통합 스크립트 사용 아래 명령어들은 run_rag_pipeline.sh 하나로도 모두 실행 가능해요. 먼저 각 단계를 이해한 뒤 스크립트로 자동화하는 것을 추천합니다.
1
🔍 전제조건 체크 파이프라인 실행 전 서비스 상태 확인
$ curl http://localhost:9200
OpenSearch가 응답하면 JSON이 출력돼요
$ curl http://localhost:9200/_cat/indices | grep physical_rag
physical_rag 인덱스가 있으면 이름이 표시돼요
$ curl http://192.168.0.23:11434/api/tags | head -5 # 임베딩관 $ curl http://192.168.0.75:11434/api/tags | head -5 # 준비관 $ curl http://192.168.0.21:11434/api/tags | head -5 # 생성관 $ curl http://192.168.0.36:11434/api/tags | head -5 # 검수관
4개관 모두 models 목록이 나오면 준비 완료
  1. OpenSearch(도서관) 응답 확인
  2. physical_rag(서고) 존재 확인
  3. 4개관(임베딩·준비·생성·검수) Ollama 서비스 확인
⚠️
어느 하나라도 응답 없으면 여기서 멈추세요 다음 단계 진행 전에 서비스를 먼저 살려야 해요. Docker: docker compose ps / Ollama: 4대 PC의 서비스 상태 확인.
2
🐍 가상환경 활성화 + 폴더 이동 rag_venv를 켜고 rag_llm 폴더로 들어가기
$ source ~/rag_venv/bin/activate $ cd ~/rag_llm
프롬프트 앞에 (rag_venv)가 붙으면 성공
  1. RAG 전용 가상환경 켜기 (lerobot과 절대 섞지 않기)
  2. 스크립트가 있는 rag_llm 폴더로 이동
💡
비유: 작업복 갈아입기 가상환경은 작업복이에요. RAG용 작업복(rag_venv)과 로봇용 작업복(lerobot)을 따로 두면 서로 안 섞여요.
3
📄 Step 1a: HTML 처리 HTML 파일을 사이드카 JSON으로 변환
$ python 01_html_processor_v4.py . --dry-run # 미리보기 $ python 01_html_processor_v4.py . # 실제 실행
각 HTML 옆에 {파일명}.rag.json 사이드카가 생겨요
▼ 예상 출력 🔍 루트: /home/zetabank/rag_llm 재귀 탐색 결과: 30개 HTML 발견 ════════════════════════════════════════════════════ 📄 [lekiwi_teleoperation] guide.html 카테고리: lekiwi | 섹션: 12개 ✅ 저장 완료 + 사이드카: guide.html.rag.json 📊 프로젝트별 처리 통계 📁 dongle_multinet_setup [network ] HTML 2개 · 섹션 8개 📁 feetech_motor_system [hardware] HTML 1개 · 섹션 25개 ... ✅ 전체 완료 — HTML 30개 · 섹션 318개
  1. rag_llm 아래 모든 HTML을 재귀 탐색
  2. 각 HTML을 파싱해 헤딩·앵커·프래그먼트 추출
  3. 원본 옆에 {파일명}.rag.json 파일 생성
📄
--dry-run으로 먼저 확인 실제 저장 없이 몇 개가 처리될지, 어떻게 분류될지 미리 볼 수 있어요. 결과가 만족스러우면 그때 --dry-run 없이 실행하세요.
4
📝 Step 1b: MD 처리 MD 파일을 사이드카 JSON으로 변환 (HTML과 같은 형식)
$ python 03_md_processor.py . --dry-run # 미리보기 $ python 03_md_processor.py . # 실제 실행
MD 파일 옆에도 {파일명}.rag.json 사이드카가 생겨요
▼ 예상 출력 🔍 루트: /home/zetabank/rag_llm MD 파일 발견: N개 ════════════════════════════════════════════════════ 📄 [knowledge] rag_lekiwi_calibration.md 카테고리: rag | 섹션: 4개 ✅ 저장: rag_lekiwi_calibration.md.rag.json 📊 프로젝트별 MD 처리 통계 📁 knowledge [rag] MD N개 · 섹션 M개 ✅ 전체 완료 — MD N개 · 섹션 M개
  1. knowledge/ 폴더 등 MD 파일 재귀 탐색
  2. YAML frontmatter(--- 사이) 파싱해 메타데이터 추출
  3. ## ### 헤딩 기준으로 섹션 분할
  4. HTML과 똑같은 사이드카 JSON 형식으로 저장
🎯
왜 같은 사이드카 형식으로? 다음 단계 인덱서는 사이드카 파일이 HTML에서 왔는지 MD에서 왔는지 상관 안 해요. 같은 형식으로 통일해두면 인덱서 하나로 둘 다 처리됩니다.
5
🏦 Step 2: OpenSearch 인덱싱 사이드카 JSON을 벡터화하며 OpenSearch에 업로드
$ python 02_indexer.py . --dry-run # 미리보기 $ python 02_indexer.py . # 실제 인덱싱 (45~60초)
임베딩관에서 벡터를 받아 OpenSearch에 100개씩 배치 업로드해요
▼ 예상 출력 🔍 사전 서버 검증... ✅ OpenSearch OK — 인덱스 'physical_rag' 확인됨 ✅ 임베딩관 OK — 모델 'bge-m3' 확인됨 🔍 사이드카 JSON 발견: (30 + N)개 📄 [lekiwi_teleoperation] guide.html.rag.json (12개 섹션) 진행: 누적 성공 12 / 실패 0 📤 배치 업로드: 성공 100 / 실패 0 ... 📊 프로젝트별 인덱싱 결과 📁 dongle_multinet_setup 섹션 8 · 성공 8 · 실패 0 📁 feetech_motor_system 섹션 25 · 성공 25 · 실패 0 ... ✅ 인덱싱 완료 · 전체 섹션: (318 + M) · 성공: (318 + M) · 실패: 0 · 소요 시간: 약 45~60초 📈 OpenSearch 최종 확인: · physical_rag 총 문서 수: (318 + M)개
  1. 사전 검증(OpenSearch, 임베딩관 응답 확인)
  2. 모든 *.rag.json 재귀 탐색
  3. 각 섹션을 임베딩관(3070)에 보내 1024차원 벡터로 변환
  4. 메타 + 벡터를 하나의 문서로 조립
  5. 100개씩 묶어 Bulk API로 OpenSearch에 업로드
🚚
Bulk = 컨테이너 배송 하나씩 300번 왕복하지 않고 100개씩 3번 왕복해요. 10~20배 빠릅니다.
⏱️
재실행해도 안전 (덮어쓰기) 같은 자료를 다시 인덱싱하면 doc_id가 같아서 덮어쓰기 됩니다. 중복 문서 안 쌓여요. 다만 45~60초는 다시 걸립니다 (증분 인덱싱은 규모 커지면 도입).
6
🔍 Step 3: 검색 테스트 하이브리드 검색(BM25 + kNN) 결과 확인
$ python 03_search.py "청킹" $ python 03_search.py "캘리브레이션" $ python 03_search.py # 대화형 모드
관련도 순으로 5개 결과와 이중 조준 URL이 나와요
▼ 예상 출력 🔍 '청킹' 검색 결과 — 5건 ══════════════════════════════════════════════════════════════════════ [1] 점수: 3.542 | [lekiwi] lekiwi_teleoperation 📄 🧬 청크와 임베딩 🔖 anchor: sec-5-청크와-임베딩 📝 미리보기: RAG 시스템에서 <em>청킹</em>은 문서를 의미 단위로 나누는 과정... 🔗 http://192.168.0.75:8000/lekiwi_teleoperation/html/guide.html#sec-5-청크와-임베딩:~:text=청크와%20임베딩 [2] 점수: 2.891 | [rag] knowledge 📄 Q3. 청킹 크기 결정 방법 ...
  1. 질문을 임베딩관에 보내 벡터로 변환
  2. OpenSearch에서 BM25(단어) + kNN(의미) 동시 검색
  3. 상위 5개 결과에 highlight 미리보기 추가
  4. relative_path + anchor + fragment 조합으로 이중 조준 URL 생성
🎯
이중 조준 URL 브라우저에서 링크 클릭하면 정확한 섹션까지 스크롤되고 검색어에 노란 형광펜이 자동으로 그어져요. 파일 서버(python3 -m http.server 8000)가 켜져있어야 브라우저로 열 수 있어요.
7
📦 확인 명령: 사이드카 JSON 개수 Step 3·4 사이에 사이드카가 잘 만들어졌는지 검증
$ find ~/rag_llm -name "*.rag.json" | head -10 # 샘플 10개 $ find ~/rag_llm -name "*.rag.json" | wc -l # 총 개수
사이드카 파일 목록과 총 개수가 나와요
▼ 예상 출력 /home/zetabank/rag_llm/lekiwi_teleoperation/html/guide.html.rag.json /home/zetabank/rag_llm/feetech_motor_system/html/motor_ref.html.rag.json /home/zetabank/rag_llm/knowledge/rag_lekiwi_calibration.md.rag.json ... 30 # HTML 30개 처리했으면 30, MD 추가되면 더 많이
  1. rag_llm 폴더 아래 *.rag.json 파일 검색
  2. 샘플 10개로 어느 폴더에 있는지 확인
  3. 총 개수 세기 → HTML+MD 합계와 맞는지 확인
8
📈 확인 명령: OpenSearch 문서 수 Step 5 이후 실제 인덱스에 들어간 문서 개수 확인
$ curl http://localhost:9200/physical_rag/_count
JSON으로 총 문서 개수(count)가 나와요
▼ 예상 출력 { "count": 318, "_shards": { "total": 1, "successful": 1, "skipped": 0, "failed": 0 } }
  1. OpenSearch에 _count API로 문서 총 개수 질의
  2. 인덱싱 결과 리포트의 성공 개수와 일치 확인
  3. 만약 다르면 실패 로그를 다시 살펴보기
🔍
더 자세한 확인 (프로젝트별) 아래 명령어로 프로젝트별 문서 수도 볼 수 있어요.
$ curl -X POST http://localhost:9200/physical_rag/_search \ -H "Content-Type: application/json" \ -d '{"size":0,"aggs":{"by_project":{"terms":{"field":"project","size":20}}}}'

🎯 추천 실행 순서 — 처음부터 끝까지

전체를 처음 실행할 때는 이 순서로 하세요.

🚀 One-shot 실행 순서

1
전제조건 체크 — bash run_rag_pipeline.sh check
2
환경 활성화 — source ~/rag_venv/bin/activate && cd ~/rag_llm
3
Dry-run 미리보기 — bash run_rag_pipeline.sh dry
4
전체 실행 — bash run_rag_pipeline.sh full
5
결과 확인 — bash run_rag_pipeline.sh count
6
추가 검색 테스트 — bash run_rag_pipeline.sh search "다른 질문"

🔧 개별 단계만 실행하고 싶을 때

# 새 HTML만 추가했을 때 $ bash run_rag_pipeline.sh html $ bash run_rag_pipeline.sh index # 새 MD만 추가했을 때 $ bash run_rag_pipeline.sh md $ bash run_rag_pipeline.sh index # 검색만 반복 테스트 $ bash run_rag_pipeline.sh search "질문 1" $ bash run_rag_pipeline.sh search "질문 2"
🎁
새 자료 추가 시 팁 특정 폴더만 처리하려면 스크립트 인자를 폴더 경로로 주면 돼요. 예: python 01_html_processor_v4.py ~/rag_llm/kiosk_setup

🔧 문제 해결 — 자주 겪는 상황

❓ Step 5(인덱싱)에서 "인덱스가 없습니다" 에러가 나요

physical_rag 인덱스를 아직 안 만드셨어요. 먼저 create_index.sh를 실행하세요.

$ bash ~/opensearch/create_index.sh http://localhost:9200

❓ 임베딩관 응답 없음 에러

임베딩관(192.168.0.23) PC의 Ollama 서비스가 꺼져있어요.

# 임베딩관 PC에 SSH로 접속 후 $ sudo systemctl status ollama $ sudo systemctl restart ollama

❓ 검색 결과가 이상해요 (관련 없는 내용이 나옴)

여러 원인이 있을 수 있어요. 아래 순서로 확인:

  1. OpenSearch 문서 수가 예상대로인지 확인 (bash run_rag_pipeline.sh count)
  2. 사이드카 JSON을 열어서 title/preview가 잘 뽑혔는지 확인
  3. 그 주제 자료가 실제로 있는지 확인 (없으면 자료 추가 필요)
  4. Nori 분석 결과 확인 (curl로 _analyze API 호출)
🚫
절대 하면 안 되는 것 • rag_venv 안에서 lerobot 라이브러리 설치 (섞이면 로봇이 안 도는 사고 발생)
• 실행 중인 인덱싱을 Ctrl+C로 중간에 끊기 (부분 인덱싱 상태로 남음)
• OpenSearch 데이터 폴더(./data)를 삭제 (모든 인덱스 소실)
